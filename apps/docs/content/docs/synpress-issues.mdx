---

Findings from getting `@synthetixio/synpress@4.1.2` (the latest published release) working with a Vite
dapp, testing MetaMask (EVM) and Phantom (EVM and Solana). This is the evidence behind building
walletwright instead; see [Background](/background). The fork and the `synpress-linux-repro.log`
referenced below live in the original `test-synpress` exploration project. walletwright is the
clean-room reimplementation of the fixes.

## TL;DR

- A direct Linux reproduction. Stock Synpress 4.1.2 ran in the official Playwright Linux container
  (`mcr.microsoft.com/playwright:v1.48.2-jammy`, `xvfb`, Synpress's exact pinned versions:
  `playwright-core` 1.48.2, Chromium 1140, MetaMask 13.13.1), the same environment as their CI. From
  the run (full log in `synpress-linux-repro.log`): the Phantom download URL is dead
  (`curl: (6) Could not resolve host: crx-backup.phantom.dev`), the MetaMask cache build succeeded,
  and `connectToDapp()` failed with `[getNotificationPageAndWaitForLoad] Notification page did not
appear after 10000ms and 2 retries`. The connect flow is broken on Linux, on Synpress's own pinned
  stack.
- Synpress's own E2E CI is red at the `release: 4.1.2` commit (`.github/workflows/test.yml`,
  `ubuntu-latest` with `xvfb-run`). Both the Metamask and Phantom headful jobs fail; only the unit
  tests pass. Run: [#20885224202](https://github.com/Synthetixio/synpress/actions/runs/20885224202)
- Open bug reports from other users match (Ubuntu, macOS Sonoma, macOS Sequoia).
- The Linux repro narrows the scope. The cache build works with the pinned versions, so
  cache-creation failures come from a Playwright/Chromium version mismatch (#1290, #1283), not from
  the pinned stack itself. The connect flow fails even on the pinned stack, so that family (#3, #4,
  #5) is the universal breakage that hits everything, including CI.

Legend: "Universal" reproduces everywhere, including Synpress's CI. "Newer-deps" appears once you use
a current Playwright, Chromium, or MetaMask (which `npx playwright install` pulls; see #2).

## 1. Phantom download URL is dead (Universal)

`packages/cache/src/prepareExtensionPhantom.ts` and `wallets/phantom/src/prepareExtensionPhantom.ts`
download from `https://crx-backup.phantom.dev/latest.crx`. That host no longer resolves (NXDOMAIN), so
every Phantom cache build or test fails immediately, everywhere. It's DNS, so it's identical on all
machines and on CI (the Phantom CI job is red).

Fix: fetch from the Chrome Web Store update endpoint (extension id `bfnaelmomeimhlpmgjnjophhpkkoljpa`).

## 2. Chromium version isn't aligned with the installed Playwright (Universal)

Synpress depends on `playwright-core@1.48.2` (Chromium build 1140) but doesn't constrain the
`@playwright/test` you install. With a current Playwright, `npx playwright install` fetches a different
Chromium revision than Synpress's cache and launch logic expect, and cache creation fails.

- issue #1283, "playwright install installs chromium 1161, synpress expects 1140"
- issue #1290, "cache creation fails… downgrade to 1.48.2 to work"

Fix: align the whole pipeline on one Playwright version (we used 1.61 with Chromium 1228).

## 3. Connect/sign popup never detected, a `waitForEvent` race (Universal, reproduced on Linux)

`wallets/metamask/.../utils/getNotificationPageAndWaitForLoad.ts` uses
`context.waitForEvent('page', {predicate: url.includes('notification.html')})`. The popup is created
as `about:blank` and then navigates to `notification.html`, so the predicate is false at creation, the
event is missed, and you get "Notification page did not appear" even though the popup is open.

Reproduced directly: stock Synpress 4.1.2 in the Playwright Linux container with the pinned
Chromium 1140 and MetaMask 13.13.1 fails here with exactly this error (`synpress-linux-repro.log`).

Fix: poll `context.pages()` for the URL instead of relying on the creation-time predicate.

## 4. `connectToDapp()` confirm-button selector is stale (Universal)

`selectors/pages/NotificationPage/actionFooter.ts` uses `page-container-footer-next`, but the shipped
MetaMask connect popup uses `confirm-btn` (and signing uses `confirm-footer-button`).

Confirmed by issue #1310, "connectToDapp() fails, confirmActionButton selector does not match
MetaMask v13.13.1's Connect button" (v13.13.1 is Synpress's own pinned version, the latest Synpress).

Fix: match `confirm-btn` for connect and `confirm-footer-button` for signing, keeping the legacy id as
a fallback.

## 5. `connectToDapp()` double-clicks the confirm button (Universal)

`actions/connectToDapp.ts` clicks the confirm button twice, for the old two-step Next then Connect
flow. The current permissions-style popup is single-step, so the second click errors or hits a closed
popup.

Fix: click once, and click again only if a confirm button is still visible.

## 6. Cached wallet stuck on "Your wallet is ready" (completedOnboarding) (Universal)

After `importWallet`, the wallet sits on the completion screen with `completedOnboarding=false`.
Acknowledging it ("Open wallet") goes through the MV3 service worker and hangs under automation, so the
cached wallet boots back to onboarding and ignores dapp requests.

Confirmed by issue #1308, "MetaMask v13 setup hangs on 'Your wallet is ready' screen", reported on
Ubuntu. Also #1307, "Hanging on metamask login".

Fix (what worked): write `completedOnboarding=true` directly into the cached profile's LevelDB (the
`OnboardingController` key on 13.3x; a single `data` blob on 13.13.x).

## 7. `networkidle` and loading-indicator waits hang (Universal)

`utils/waitFor.ts` `waitUntilStable()` blocks on `waitForLoadState('networkidle')`. MetaMask popups
keep background connections open and never reach networkidle, so it times out and aborts
otherwise-ready interactions. `waitForMetaMaskLoad()` similarly blocks on loading indicators that can
stay visible.

Fix: make networkidle best-effort (catch the timeout) and don't hard-block on every loading indicator.

## 8. MV3 service-worker instability under automation (Universal, flaky)

MetaMask intermittently throws "Background 'getState' call exceeded timeout" and the wallet UI crashes
during the fixture flow; the fixture's own crash-recovery reloads can make it worse. It shows up as
the flaky or failing E2E on CI. Related: issue #1303 (MV3 incompatibility with the latest Chromium) and
#1282 (`approveTokenPermission` flaky).

Mitigation: keep the service worker warm with a lean flow, run headed, and add retries.

## 9. Headless breaks extension approval popups (Universal, a constraint)

Connect and sign popups only open in headed mode on current Chromium. Synpress's own CI works around
this with `xvfb-run` (a virtual display). Worth documenting, because the defaults invite a headless CI.

## 10. Phantom "You're all ready!" onboarding selector is stale (Universal)

In `selectors/pages/OnboardingPage`, the `allDone` text no longer exists (the final screen is a "Get
Started" button), so Phantom `importWallet` fails its completion assertion even though the import
succeeded. The Phantom CI job is red.

## 11. Phantom blocks the documented test seed (Universal)

Phantom flags the well-known public seed `test test … junk` (used throughout the Synpress docs and
examples) as "might be malicious" and silently drops dapp connections (no popup, the request hangs
forever). It isn't Synpress's code, but it makes the Phantom examples look broken.

Fix: use a fresh, non-famous mnemonic. Related: #1289, "Metamask on solana testing".

## Issues triggered once you use current Chromium or MetaMask (Newer-deps)

These don't appear on Synpress's exact pins, but show up the moment you use a current Playwright (which
#2 forces) or bump MetaMask:

- **12. `--load-extension` missing.** `metaMaskFixtures.ts` and `phantomFixtures.ts` launch with only
  `--disable-extensions-except` (the comment reads "we don't need `--load-extension`… already loaded
  in the cache"). On current Chromium the cached unpacked extension does not auto-load, so you get
  fresh onboarding.
- **13. No MV3 service-worker wait.** Fixtures interact before the worker restores the vault, so the UI
  shows onboarding instead of unlock. It needs `waitForEvent('serviceworker')` plus reload-until-unlock.
- **14. Outdated onboarding selectors (MetaMask 13.3x).** There's a new `setup-passkey` step, the
  password testids changed to `create-password-new-input` and `create-password-confirm-input`, and the
  metrics step is `metametrics-i-agree`. `importWallet` doesn't handle these.
- **15. `MetaMask.unlock()` uses the `unlock-password` testid**, which is absent in MetaMask 13.35.x
  (it's a plain password input now).

## How this was verified

- Direct Linux container repro: stock Synpress 4.1.2 in `mcr.microsoft.com/playwright:v1.48.2-jammy`
  with `xvfb` and pinned Chromium 1140 / MetaMask 13.13.1. The Phantom URL was dead, the cache build
  was fine, and `connectToDapp()` failed ("Notification page did not appear"). Full log:
  `synpress-linux-repro.log`. Command used:
  `docker run --rm --cap-add=SYS_ADMIN --ipc=host -v <repro>:/work -w /work mcr.microsoft.com/playwright:v1.48.2-jammy bash run.sh`.
- Synpress CI: `gh run list -R Synthetixio/synpress -w test.yml` shows red at 4.1.2; the jobs run on
  `ubuntu-latest` with `xvfb-run`.
- Issue tracker: `gh issue list -R Synthetixio/synpress` shows the bugs above (#1310, #1308, #1307,
  #1303, #1290, #1289, #1283, #1282, and more), reported on Ubuntu and macOS Sonoma/Sequoia by
  different users.
- Fork with fixes: branch `fix/synpress-4.1.2-compat` (15 files changed) carries a concrete diff for
  every issue above. `npm test`, `npm run demo`, and `npm run demo:phantom` all pass against it
  (MetaMask 13.35.1, Phantom v26.17.0, Playwright 1.61, headed).
