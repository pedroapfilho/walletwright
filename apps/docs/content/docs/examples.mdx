Every recipe below uses the same `wallet` fixture API. The wallet-side calls (`connectToDapp()`,
`confirmSignature()`, `approve()`) are identical across MetaMask, Phantom, and Slush; only your
dapp-side selectors change. The `getByTestId(...)` calls here are your dapp's own elements, not
walletwright's.

## A shared wallet-setup module

Keep every `WalletSetup` in one module so the cache build and your specs import the same objects. A
change to `wallet`, `seedPhrase`, `password`, or `version` produces a distinct cached profile.

```ts
// wallet-setup.ts
import type { WalletSetup } from "walletwright";

export const metamask: WalletSetup = {
  wallet: "metamask",
  seedPhrase: "test test test test test test test test test test test junk",
  password: "Tester@1234",
};

// Phantom and Slush flag the famous public seed as malicious and drop the connection.
// Use a fresh, unfunded mnemonic for both.
export const phantom: WalletSetup = {
  wallet: "phantom",
  seedPhrase: "parade wire fork giggle foil sugar early record crew mesh excuse purse",
  password: "Tester@1234",
};

export const slush: WalletSetup = {
  wallet: "slush",
  seedPhrase: "parade wire fork giggle foil sugar early record crew mesh excuse purse",
  password: "Tester@1234",
};
```

## Connect and sign (MetaMask)

The core loop: click your dapp's button, then approve the wallet popup. `connectToDapp()` clears the
connection popup; `confirmSignature()` clears the signature popup.

```ts
// tests/metamask.spec.ts
import { createWalletFixtures } from "walletwright";
import { metamask } from "../wallet-setup";

const test = createWalletFixtures(metamask);
const { expect } = test;

// The first account of the public test seed is the well-known Hardhat address.
const ACCOUNT = "0xf39fd6e51aad88f6f4ce6ab8827279cfffb92266";

test("connect and sign", async ({ page, wallet }) => {
  await page.goto("/");

  await page.getByTestId("connect").click();
  await wallet.connectToDapp();
  await expect(page.getByTestId("account")).toHaveText(ACCOUNT);

  await page.getByTestId("sign").click();
  await wallet.confirmSignature();
  await expect(page.getByTestId("signature")).toHaveText(/^0x[0-9a-fA-F]{130}$/);
});
```

## Connecting through a connector library (wagmi, RainbowKit, ...)

Most real dapps connect through a library rather than calling `window.ethereum` directly. The
wallet-side calls do not change: click the library's connect button, pick the wallet in its modal,
then approve. Because walletwright loads the real MetaMask extension, the library's EIP-6963
discovery finds it like any user's wallet.

```ts
// tests/rainbowkit.spec.ts
import { createWalletFixtures } from "walletwright";
import { metamask } from "../wallet-setup";

const test = createWalletFixtures(metamask);
const { expect } = test;

test("connect through the RainbowKit modal", async ({ page, wallet }) => {
  await page.goto("/");

  await page.getByRole("button", { name: "Connect Wallet" }).click();
  await page.getByRole("button", { name: "MetaMask" }).click(); // the connector's modal
  await wallet.connectToDapp();

  await expect(page.getByText(/0x[0-9a-fA-F]{4}/)).toBeVisible(); // the truncated address in the UI
});
```

<Callout type="info">
  The same shape works for wagmi, ConnectKit, Web3Modal / Reown AppKit, Dynamic, and Privy: they all
  end at the injected provider (or EIP-6963) and the extension's approval popup. Only the modal's
  button label changes. For a Privy external-wallet flow, its login modal also raises a SIWE
  `personal_sign` right after connect, so approve twice: `connectToDapp()` then
  `confirmSignature()`.
</Callout>

## One test, two chains (Phantom EVM + Solana)

Phantom drives both `window.phantom.ethereum` and `window.phantom.solana`. The same two calls handle
each chain, so a single spec can connect and sign on EVM and then on Solana.

```ts
// tests/phantom.spec.ts
import { createWalletFixtures } from "walletwright";
import { phantom } from "../wallet-setup";

const test = createWalletFixtures(phantom);
const { expect } = test;

test("phantom EVM + Solana", async ({ page, wallet }) => {
  await page.goto("/");

  // EVM (window.phantom.ethereum)
  await page.getByTestId("phantom-evm-connect").click();
  await wallet.connectToDapp();
  await page.getByTestId("phantom-evm-sign").click();
  await wallet.confirmSignature();

  // Solana (window.phantom.solana)
  await page.getByTestId("phantom-svm-connect").click();
  await wallet.connectToDapp(); // may auto-approve a site Phantom already trusts
  await page.getByTestId("phantom-svm-sign").click();
  await wallet.confirmSignature();
});
```

<Callout type="info">
  `connectToDapp()` resolves quietly when no popup appears, because a wallet may auto-approve a site
  it already trusts on a second chain. `confirmSignature()` requires a popup and throws if none
  shows up.
</Callout>

## Sui (Slush)

Slush connects through the Wallet Standard and re-prompts for the password when signing. The fixture
handles the re-auth, so the test is still just connect then sign.

```ts
// tests/slush.spec.ts
import { createWalletFixtures } from "walletwright";
import { slush } from "../wallet-setup";

const test = createWalletFixtures(slush);
const { expect } = test;

test("connect and sign on Sui", async ({ page, wallet }) => {
  await page.goto("/");

  await page.getByTestId("sui-connect").click();
  await wallet.connectToDapp();
  await expect(page.getByTestId("sui-account")).toHaveText(/^0x[0-9a-fA-F]+$/);

  await page.getByTestId("sui-sign").click();
  await wallet.confirmSignature();
  await expect(page.getByTestId("sui-signature")).not.toBeEmpty();
});
```

## Privy with external wallets

A dapp using [Privy](https://privy.io) with external wallets enabled doesn't change what walletwright
drives: Privy talks to the same injected provider, and the extension opens its usual popups. Your
test clicks through Privy's login modal (it renders in the page's DOM, so plain locators work), and
the wallet calls stay the same. The one Privy-specific detail is that it authenticates external
wallets with a SIWE message, so a connect popup and a `personal_sign` popup arrive back to back;
the login isn't complete until both are approved.

```tsx
// The dapp side: external wallets only, no embedded wallet.
<PrivyProvider
  appId={appId}
  config={{
    appearance: { walletList: ["detected_ethereum_wallets"] },
    embeddedWallets: { ethereum: { createOnLogin: "off" } },
    loginMethods: ["wallet"],
  }}
>
  <Dapp />
</PrivyProvider>
```

```ts
// tests/privy.spec.ts
import { createWalletFixtures } from "walletwright";
import { metamask } from "../wallet-setup";

const test = createWalletFixtures(metamask);
const { expect } = test;

test("log in with Privy via MetaMask", async ({ page, wallet }) => {
  await page.goto("/");

  await page.getByRole("button", { name: "Log in" }).click();
  await page.getByRole("button", { name: /MetaMask/i }).click(); // Privy's wallet list

  await wallet.connectToDapp(); // the connect popup
  await wallet.confirmSignature(); // Privy's SIWE personal_sign popup

  await expect(page.getByTestId("account")).toHaveText(/^0x[0-9a-fA-F]{40}$/);
});
```

<Callout type="warn">
  This covers Privy's external-wallet path only. Embedded wallets (email, social, passkey login)
  keep their keys inside Privy's own iframe: there is no extension popup, so there is nothing for
  walletwright to drive.
</Callout>

You need a Privy **app id** (the public client-side identifier from
[dashboard.privy.io](https://dashboard.privy.io), no app secret) with wallet login enabled and your
dapp's origin allowed. The demo page falls back to Privy's shared public test app id, which is fine
for clicking the flow by hand but too rate-limited to drive a reliable automated run, so the spec
runs only when you provide your own id, either as an env var or in a local gitignored
`apps/demo/.env` (see `.env.example`; the Playwright config loads it):

```bash
pnpm test:cache
VITE_PRIVY_APP_ID=your-app-id pnpm test tests/privy.spec.ts
```

<Callout type="info">
  Because Privy's connect and SIWE popups can arrive with variable delay, the demo spec drives
  whatever approval popup is pending until the account appears, rather than assuming a fixed
  connect-then-sign order. See `apps/demo/tests/privy.spec.ts`.
</Callout>

The demo app has a runnable version of this recipe (`apps/demo/privy.html` and
`tests/privy.spec.ts`).

## Rejecting a request

Every approval has a counterpart: `rejectConnection()`, `rejectSignature()`, and the generic
`reject()`. Rejection is what your dapp's unhappy path is made of, and it is worth testing, because
the wallet rejects the provider promise with an EIP-1193 error object (`{ code: 4001, message }`)
rather than an `Error`. Read `error.message`, or you will render `[object Object]`.

```ts
test("shows an error when the user rejects the signature", async ({ page, wallet }) => {
  await page.goto("/");

  await page.getByTestId("connect").click();
  await wallet.connectToDapp();

  await page.getByTestId("sign").click();
  await wallet.rejectSignature();

  await expect(page.getByTestId("error")).toContainText(/reject/i);
});
```

## Locking the wallet

`settings.lock()` locks the wallet from its own UI, which is how you test that your dapp handles a
wallet that locked mid-session. `settings.unlock()` reverses it with the password from your setup.

```ts
test("prompts to unlock after the wallet locks", async ({ page, wallet }) => {
  await page.goto("/");
  await page.getByTestId("connect").click();
  await wallet.connectToDapp();

  await wallet.settings.lock();
  // …assert your dapp's locked-wallet handling here
  await wallet.settings.unlock();
});
```

<Callout type="info">
  Capabilities are per-wallet. A wallet declares an action only once it has been driven against the
  real extension, so calling one it hasn't declared throws (`[walletwright] Slush does not support
  reject()`) instead of quietly doing nothing. `reject` is verified on MetaMask and Phantom;
  `settings` on MetaMask.
</Callout>

## Networks and a local chain

Network and transaction tests need a chain the wallet can actually reach: MetaMask validates a
custom network's RPC before saving it, and reads balance and nonce off the chain to render its
popups. `walletwright/chain` boots a local node (Foundry's `anvil` via `prool`) whose default
mnemonic pre-funds the same account the MetaMask setup imports.

```ts
// global-setup.ts
import { createLocalChain } from "walletwright/chain";

const chain = await createLocalChain(); // http://127.0.0.1:8545, chain id 31337
```

`network.add` drives the wallet's own add-network form:

```ts
await wallet.network.add({
  name: "Walletwright Local",
  chainId: 31337,
  rpcUrl: "http://127.0.0.1:8545",
  symbol: "ETH",
});
```

Switching is different. MetaMask 13.x scopes the active chain per dapp and has no wallet-side
selector anymore, so the dapp initiates the switch and the wallet approves it. Have the dapp call
`wallet_addEthereumChain` rather than a bare EIP-3326 `wallet_switchEthereumChain`: the add is
idempotent (it adds when missing and switches when present), and in MetaMask 13.x a bare switch
request to a wallet-added custom chain hangs with no popup and no error.

```ts
test("switch to the local chain", async ({ page, wallet }) => {
  await page.goto("/");
  await page.getByTestId("connect").click();
  await wallet.connectToDapp();

  // Your dapp calls wallet_addEthereumChain with the local chain's config on this click.
  await page.getByTestId("switch-chain").click();
  // MetaMask confirms the add and the switch in a single popup. The first request routed through
  // a protocol Snap also shows a "Third-party software notice", which approve() handles.
  await wallet.approve();

  await expect(page.getByTestId("chain-id")).toHaveText("0x7a69");
});
```

`wallet.network.add` is for the cases where the network must exist in the wallet before the dapp
asks for it, e.g. testing how your dapp behaves when the user already has the chain configured. The
runnable version of both flows is `apps/demo/tests/metamask-network.spec.ts`.

## Managing accounts

`accounts` drives MetaMask's account list: `add()` derives the next HD account, `switch(index)`
makes one active, `rename({ index, name })` renames it, and `importPrivateKey(key)` adds an account
from a raw key. Indexes follow the order shown in the wallet's own list.

```ts
test("second account signs", async ({ page, wallet }) => {
  await wallet.accounts.add(); // derive Account 2 from the seed
  await wallet.accounts.switch(1); // make it active

  await page.goto("/");
  await page.getByTestId("connect").click();
  await wallet.connectToDapp();
  // …the dapp now sees the second account's address
});
```

`importPrivateKey` is handy for funding a known account on a local chain:

```ts
await wallet.accounts.importPrivateKey(
  "0xac0974bec39a17e36ba4a6b4d238ff944bacb478cbed5efcae784d7bf4f2ff80",
);
```

## Sending a transaction

Transactions need a chain where the sender holds funds; combine this with the local chain from the
previous recipe. `confirmTransaction()` approves the popup, `rejectTransaction()` cancels it, and
the generic `approve()`/`reject()` still work when you don't care about the request type.

```ts
test("send a transaction", async ({ page, wallet }) => {
  await page.goto("/");

  await page.getByTestId("connect").click();
  await wallet.connectToDapp();
  await page.getByTestId("switch-chain").click();
  await wallet.approve();

  await page.getByTestId("send-tx").click();
  await wallet.confirmTransaction();

  await expect(page.getByTestId("tx-hash")).toHaveText(/^0x[0-9a-fA-F]{64}$/);
});
```

The runnable version, including asserting the receipt on the local chain and the rejection path, is
`apps/demo/tests/metamask-transaction.spec.ts`.

<Callout type="info">
  Pass `approve({ optional: true })` when a popup may or may not appear (for example a second
  approval the wallet already trusts). It resolves quietly instead of throwing when none is pending.
</Callout>

## Rejecting a transaction

The reject counterparts mirror the confirmations. `rejectTransaction()` cancels the transaction
popup, and the dapp receives the EIP-1193 rejection (`{ code: 4001 }`).

```ts
test("rejecting a transaction surfaces an error", async ({ page, wallet }) => {
  await page.goto("/");
  await page.getByTestId("connect").click();
  await wallet.connectToDapp();

  await page.getByTestId("send-tx").click();
  await wallet.rejectTransaction();

  await expect(page.getByTestId("error")).toContainText(/reject|denied/i);
  await expect(page.getByTestId("tx-hash")).toBeEmpty();
});
```

## Asserting on-chain state with viem

Approving a popup proves the wallet signed, not that the transaction landed. Pair
`walletwright/chain` with [viem](https://viem.sh) to read the chain directly and assert the receipt.
Putting the chain in a fixture starts it with the suite and stops it after.

```ts
// tests/onchain.spec.ts
import { createWalletFixtures } from "walletwright";
import { createLocalChain, type LocalChain } from "walletwright/chain";
import { createPublicClient, http } from "viem";
import { metamask } from "../wallet-setup";

const test = createWalletFixtures(metamask).extend<{ chain: LocalChain }>({
  chain: async ({}, use) => {
    const chain = await createLocalChain();
    await use(chain);
    await chain.stop();
  },
});
const { expect } = test;

test("the transfer is mined", async ({ page, wallet, chain }) => {
  const client = createPublicClient({ transport: http(chain.rpcUrl) });

  await page.goto("/");
  await page.getByTestId("connect").click();
  await wallet.connectToDapp();
  await page.getByTestId("switch-chain").click();
  await wallet.approve();

  await page.getByTestId("send-tx").click();
  await wallet.confirmTransaction();

  const hash = (await page.getByTestId("tx-hash").textContent()) as `0x${string}`;
  const receipt = await client.waitForTransactionReceipt({ hash });
  expect(receipt.status).toBe("success");
});
```

<Callout type="warn">
  `createLocalChain` uses port 8545 by default, and a persistent-context wallet run is `workers: 1`,
  so one chain per suite is the right granularity. If you run several chain-backed suites, give each
  a distinct `port` (and point your dapp's config at it) so they don't collide.
</Callout>

## Without an extension: the mock wallet

Not every dapp test needs a real extension. When you're testing your dapp's connect-and-sign wiring,
not the wallet, `walletwright/mock` injects a headless EIP-1193 provider (announced over EIP-6963)
that signs with a real key. These specs use plain `@playwright/test`, run headless, and finish in a
second or two.

```ts
import { test, expect } from "@playwright/test";
import { installMockWallet } from "walletwright/mock";
import { verifyMessage } from "viem";

test("connect and sign, no extension", async ({ page }) => {
  const address = await installMockWallet(page.context());

  await page.goto("/");
  await page.getByTestId("connect").click();
  await expect(page.getByTestId("account")).toHaveText(new RegExp(address, "i"));

  await page.getByTestId("sign").click();
  const signature = await page.getByTestId("signature").textContent();
  expect(await verifyMessage({ address, message: "Hello", signature })).toBe(true);
});
```

The mock is a connect/sign stand-in, not an EVM: it doesn't execute transactions. When you need real
transactions, balances, or receipts, drive an actual extension against the local chain (see above).

`installMockWallet` takes options to control the account, chain, and announced name:

```ts
const address = await installMockWallet(page.context(), {
  // anvil/hardhat account #1; the mock signs with this key so signatures verify.
  privateKey: "0x59c6995e998f97a5a0044966f0945389dc9e86dae88c7a8412f4603b6b78690d",
  chainId: 1, // what the dapp sees from eth_chainId
  name: "My Test Wallet", // announced over EIP-6963
});
```

## Composing your own fixtures

`createWalletFixtures(setup)` returns a normal `@playwright/test` `test`, so extend it with your own
fixtures. Here a `connected` fixture connects the wallet once so each test starts already connected.

```ts
// tests/fixtures.ts
import { createWalletFixtures } from "walletwright";
import { metamask } from "../wallet-setup";

export const test = createWalletFixtures(metamask).extend<{ connected: void }>({
  connected: [
    async ({ page, wallet }, use) => {
      await page.goto("/");
      await page.getByTestId("connect").click();
      await wallet.connectToDapp();
      await use();
    },
    { auto: true }, // run for every test in files that import this `test`
  ],
});
export const { expect } = test;
```

```ts
// tests/signing.spec.ts
import { test, expect } from "./fixtures";

test("sign, already connected", async ({ page, wallet }) => {
  await page.getByTestId("sign").click();
  await wallet.confirmSignature();
  await expect(page.getByTestId("signature")).toHaveText(/^0x[0-9a-fA-F]{130}$/);
});
```

## A wallet page object

For a larger suite, wrap the dapp interaction and the wallet approval together so specs read at one
level of abstraction. The `wallet` fixture is just an object you can pass into a page object.

```ts
// tests/dapp.page.ts
import type { Page } from "@playwright/test";
import type { Wallet } from "walletwright";

export class DappPage {
  constructor(
    private readonly page: Page,
    private readonly wallet: Wallet,
  ) {}

  async connect(): Promise<void> {
    await this.page.goto("/");
    await this.page.getByTestId("connect").click();
    await this.wallet.connectToDapp();
  }

  async sign(message: string): Promise<string> {
    await this.page.getByTestId("message").fill(message);
    await this.page.getByTestId("sign").click();
    await this.wallet.confirmSignature();
    return (await this.page.getByTestId("signature").textContent()) ?? "";
  }
}
```

```ts
// tests/dapp.spec.ts
import { createWalletFixtures } from "walletwright";
import { metamask } from "../wallet-setup";
import { DappPage } from "./dapp.page";

const test = createWalletFixtures(metamask);
const { expect } = test;

test("connect and sign through the page object", async ({ page, wallet }) => {
  const dapp = new DappPage(page, wallet);
  await dapp.connect();
  const signature = await dapp.sign("Hello walletwright");
  expect(signature).toMatch(/^0x[0-9a-fA-F]{130}$/);
});
```

## Build the cache in Playwright global setup

Instead of a separate `walletwright cache` step, onboard every wallet from Playwright's `globalSetup`
so a single `playwright test` does everything. Cache building can run headless; the tests stay headed.

```ts
// playwright.config.ts
import { defineConfig } from "@playwright/test";

export default defineConfig({
  testDir: "./tests",
  globalSetup: "./global-setup.ts",
  retries: 2,
  workers: 1,
  use: { baseURL: "http://localhost:3000" },
  webServer: { command: "pnpm dev", url: "http://localhost:3000", reuseExistingServer: true },
});
```

```ts
// global-setup.ts
import { buildCache } from "walletwright";
import { metamask, phantom, slush } from "./wallet-setup";

const globalSetup = async () => {
  for (const setup of [metamask, phantom, slush]) {
    await buildCache(setup, { headless: true });
  }
};

export default globalSetup;
```

## Driving multiple wallets in one suite

There is no special multi-wallet runner. Give each wallet its own spec, each calling
`createWalletFixtures` with its own setup. They share the cache built above and run under the same
`workers: 1` config so browsers don't contend for the display.

```
tests/
  metamask.spec.ts   // createWalletFixtures(metamask)
  phantom.spec.ts    // createWalletFixtures(phantom)
  slush.spec.ts      // createWalletFixtures(slush)
```

## Without fixtures: `launchWallet`

For a one-off script or a custom runner, `launchWallet(setup)` returns `{ context, wallet }` directly.
You open the pages and close the context yourself.

```ts
// scripts/manual-check.ts
import { launchWallet } from "walletwright";
import { metamask } from "../wallet-setup";

const { context, wallet } = await launchWallet(metamask);
const page = await context.newPage();
await page.goto("http://localhost:3000");

await page.getByTestId("connect").click();
await wallet.connectToDapp();
console.log("connected; extension id:", wallet.extensionId);

await context.close();
```

<Callout type="warn">
  `launchWallet` needs a cache already built for the same setup (via `buildCache` or the CLI), and
  it launches headed, so run it on a machine with a display or under a virtual one such as
  `xvfb-run`.
</Callout>
